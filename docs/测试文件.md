## 场景 A：全新项目（干净，从零接入）

### A.1 创建空项目并安装 auto

```bash
mkdir -p /tmp/visual-newproj && cd /tmp/visual-newproj
npm init -y >/dev/null
npm install "$AUTO_PATH" >/dev/null 2>&1
```

**断言 1（零侵入）—— 安装完没有任何 visual 文件：**

```bash
ls -la .ai-spec 2>/dev/null || echo "✅ .ai-spec 不存在（正确）"
grep -ic visual package.json
# 期望：第 1 条 "✅ .ai-spec 不存在"
# 期望：第 2 条 "0"
```

### A.2 跑一次 protocol-status，确认无 visual 干扰

```bash
npx ai-spec-auto protocol-status
# 期望：状态正常显示（如 No active run），不出现 visual 相关警告
```

### A.3 显式开通 visual 桥接（opt-in）

非交互模式（用 `--yes` 接受默认值，再手动改一下 server URL 与 workspace_id）：

```bash
npx ai-spec-auto visual init --yes
# 输出末行会打印 connect_token，先复制保存

cat .ai-spec/visual-bridge.json
```

把 `server_url`/`workspace_id` 改一下：
```bash
node -e "
const fs=require('fs');
const f='.ai-spec/visual-bridge.json';
const j=JSON.parse(fs.readFileSync(f,'utf-8'));
j.server_url='${VISUAL_URL}';
j.workspace_id='newproj-demo';
fs.writeFileSync(f, JSON.stringify(j,null,2)+'\n');
console.log(j);
"
```

**断言 2（opt-in 边界）—— 此时仅多了 1 个 json 文件 + 1 个空 inbox 目录：**
```bash
find .ai-spec -maxdepth 2 -mindepth 1
# 期望：只看到 .ai-spec/visual-bridge.json 与 .ai-spec/inbox（空）
```

### A.4 在 visual 中建对应的 Workspace

打开浏览器访问 `http://localhost:18780/workspaces`，新建工作区时**手动指定 ID**为 `newproj-demo`（也可以走 admin SQL 直接 insert）。最简单的方法用 curl + DB 工具，或者直接进入 Prisma Studio：

```bash
# 一行命令快建 workspace（直接连 docker 内的 mariadb）
docker exec -i $(docker ps -q -f "expose=3306") \
  mariadb -uroot -proot br_ai_spec -e \
  "INSERT IGNORE INTO Workspace (id, slug, name, status, createdAt, updatedAt, rootPath) VALUES ('newproj-demo','newproj-demo','New Demo','active', NOW(), NOW(), '/tmp/visual-newproj');"
```

> 设置 `rootPath=/tmp/visual-newproj` 同时也开启了**文件兜底通道**——visual 写 outbox 时会同时落到 `/tmp/visual-newproj/.ai-spec/inbox/control-*.json`。

### A.5 跑一次桥接连通自检

```bash
cd /tmp/visual-newproj
npx ai-spec-auto visual test
# 期望输出三行：
# [visual] ping http://localhost:18780 → ok (status 200)
# [visual] pull pending → 0 written (transport=http)
# [visual] push probe receipt → ok
```

打开 `http://localhost:18780/overview`，应能看到 `newproj-demo` 出现在工作区健康卡里。

### A.6 触发一次真实事件上行

```bash
cd /tmp/visual-newproj
# 执行任意 protocol-* 命令，会同时触发 hook push + inbox 消费
npx ai-spec-auto protocol-step --user-input "visual e2e 测试-A6" 2>&1 | tail -20
```

**前往 visual UI 验证：**
- `http://localhost:18780/workspaces/newproj-demo/board` → 应看到一张新 run 卡片出现在 Backlog/Proposal 泳道
- `http://localhost:18780/workspaces/newproj-demo` → 点开 run 详情，能看到 Trace Stream 中有 `run.started` / `run.state_changed` 事件
- 顶部"打开 Trace Stream"抽屉 → 看到事件流

### A.7 控制面闭环：浏览器审批 → auto 应用 → 回执

**方法一（推荐，UI 操作）：**
- 在 Kanban 上把 run 卡片**拖**到下一个泳道（如 Backlog → Proposal） → 触发 approve
- 或在 RunDetail 右下 Gate Panel 点击「批准」

**方法二（命令行模拟）**：
```bash
# 用 connect_token（A.3 拿到的）从 bridge 文件里抓
TOKEN=$(node -e "console.log(require('/tmp/visual-newproj/.ai-spec/visual-bridge.json').connect_token)")

# 用 curl 直接调 visual 的 approve 接口（需要登录态，下面给免登录的兜底方法）
ls /tmp/visual-newproj/.ai-spec/inbox/  # 触发审批前
```

**最简单的方法：直接走文件兜底通道**——前面 A.4 给 workspace 配了 rootPath，所以 visual 任何 enqueue 都会自动落盘到 `.ai-spec/inbox/`。在 UI 上点完 approve 后，立即查看：

```bash
ls /tmp/visual-newproj/.ai-spec/inbox/
# 期望看到 control-<ts>-<id>.json 文件
cat /tmp/visual-newproj/.ai-spec/inbox/control-*.json | head -40
# 期望看到 outbox_id / command / payload / signature
```

**让 auto 消费这条控制指令：**
```bash
cd /tmp/visual-newproj
npx ai-spec-auto protocol-status        # 任一 protocol-* 命令都会触发 inbox-consumer
ls /tmp/visual-newproj/.ai-spec/inbox/   # control-*.json 应消失
ls /tmp/visual-newproj/.ai-spec/inbox/.applied/ /tmp/visual-newproj/.ai-spec/inbox/.processed/ 2>/dev/null
# 期望在 .applied 或 .processed 目录里看到原文件（带时间戳前缀）
```

**回到 visual UI：**
- `http://localhost:18780/workspaces/newproj-demo/board` → 该 run 卡上的 outbox 标记应从 `pending → delivered → applied`
- RunDetail 的 Gate Panel 历史区域显示「已应用 ✓」

### A.8 停服降级验证

```bash
# 停 visual
cd "$VISUAL_PATH" && lsof -ti:18780 | xargs kill -9
sleep 2
curl -s http://localhost:18780/api/health || echo "✅ visual 已停"

# 在 auto 项目里继续推进
cd /tmp/visual-newproj
time npx ai-spec-auto protocol-status 2>&1 | tail -5
# 期望：命令在 1-2 秒内完成，stderr 可能有 push failed 警告，但 exit code = 0
```

### A.9 卸载 visual

```bash
cd /tmp/visual-newproj
npx ai-spec-auto visual disable
# 或直接：rm .ai-spec/visual-bridge.json
ls -la .ai-spec/
# 期望：除了 inbox 历史归档，再没 visual 行为
```

## 常见问题 / 排错

- **看不到 control-*.json 落盘**：确认 visual 数据库里 `Workspace.rootPath` 已设置为目标项目路径（B.6/A.4 的 SQL）。
- **visual UI 上 outbox 一直 pending**：在 auto 项目里跑任一 `protocol-*` 命令触发消费；HTTP-pull 默认 50ms 超时，太慢的网会回退到 file 通道。
- **`visual test` 报 push failed 但 ping ok**：检查 `connect_token` 与 visual 端 `AI_SPEC_BRIDGE_SECRET` / `REALTIME_CONNECT_SECRET` 是否一致；不一致也会被"loose token"放行入库，但 Outbox 验签会拒绝。
- **停服后命令变慢**：默认超时 50ms（pull）+ 200ms（push），都是 best-effort。如发现 >2s，看 `bin/visual-bridge.js` 是否被改动。

按这份清单跑一遍，10 个断言全部通过，就等同于"端到端 + 零侵入 + 降级安全"全部成立。
